Password Hashing (bcrypt)

Passwords are never stored as plain text. They are hashed using bcrypt, which adds a random salt and applies 10+ rounds of processing.

// Simulated bcrypt hash (Node.js example) const hash = bcrypt.hash(password, 10); // Stored in DB: "$2b$10$N9qo8u..." // Original password: never recoverable
🧹

Input Validation & Sanitization

All inputs are validated before processing. Special characters are stripped to prevent SQL Injection and XSS attacks.

// Validation rules applied: username: letters, numbers only password: min 8 chars required HTML tags: <script> stripped out SQL chars: ' ; -- removed
🚫

Safe Error Messages

Login errors never reveal whether the username or password is wrong individually — this prevents user enumeration attacks.

// BAD (reveals info): "User not found" "Wrong password" // GOOD (vague, safe): "Invalid credentials"
🔑

Session Management

After login, a secure session token is generated. The token is validated on every request and destroyed on logout.

// Session token example token: "a8f3c9d1..." (random) expires: 30 minutes httpOnly: true (JS can't read) secure: true (HTTPS only)

SQL Injection — Introduction

WebGoat Lesson • Category: Injection Flaws • Difficulty: Beginner

SQL Injection (SQLi) is one of the most critical and widespread web vulnerabilities, consistently listed in the OWASP Top 10. It occurs when untrusted user input is inserted directly into a database query without proper sanitization.

How does it happen?

When a login form directly concatenates user input into an SQL query, an attacker can type SQL code as their "username" and manipulate the query logic.

⚠ Vulnerable Query (never do this)
-- User types: admin' OR '1'='1 SELECT * FROM users WHERE username = 'admin' OR '1'='1' AND password = 'anything'; -- Result: Returns ALL users! Attacker logs in as admin.

The query always evaluates to TRUE because '1'='1' is always true, bypassing authentication completely.

Common SQL Injection Payloads

These are the types of inputs attackers try in vulnerable login forms:

💀
Bypass Login: ' OR '1'='1

Makes the WHERE clause always TRUE. Logs in without any valid credentials.

💀
Comment Attack: admin'--

The -- comments out the rest of the SQL, skipping the password check entirely.

💀
Data Extraction: UNION SELECT

Attacker can extract usernames and passwords from the database by appending a UNION query.

💀
Drop Table: '; DROP TABLE users;--

Deletes entire database tables, causing catastrophic data loss.

If this app had no security: typing ' OR '1'='1 in the username field would grant instant admin access to anyone.

How this application prevents SQL Injection
⚠ Vulnerable approach
// String concatenation — DANGEROUS query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";
✓ Safe approach — Parameterized Queries
// Prepared statement — SAFE query = "SELECT * FROM users WHERE username = ? AND password = ?"; db.execute(query, [username, passwordHash]); // The database treats ? as DATA, not as code. // ' OR '1'='1 becomes a literal string — harmless!
Additional layers in this app

✓ Input Validation

  • Username: alphanumeric only
  • Password: minimum length enforced
  • Special SQL chars rejected

✓ Password Hashing

  • bcrypt with salt rounds = 10
  • Even if DB is stolen, hashes are useless
  • Timing-safe comparison used

  • SQL Injection → authentication bypass

✓ How our measures prevent attacks

  • bcrypt hashing protects even if DB leaked
  • Input validation strips malicious characters
  • Generic errors prevent user enumeration
  • Secure session tokens prevent hijacking
